# Good Objective Skill

## Context

Criteria for validating whether an OKR objective is well-defined. Apply when auditing or drafting an objective.

## Rules

**A good objective must:**
1. Be qualitative — no numbers (numbers belong in KRs).
2. Be specific — not generic or vague ("improve", "optimize", "grow" are red flags).
3. Define a clear business change — not a task or a solution.
4. Have real impact: revenue, users, retention, or competitive advantage.
5. Be uncomfortable — it forces a change in how you operate.
6. Be time-bound — scoped to a period (e.g., 90 days / Q).
7. Serve as direction, not a list of actions.
8. Allow measurable KRs to be derived from it easily.
9. Be a priority — maximum 1–3 objectives at a time.

**Red flags (bad objective):**
- Contains numbers → that is a KR, not an objective.
- Is a task ("launch", "create", "implement").
- Is vague ("improve", "optimize", "grow").
- Changes nothing relevant if achieved.
- Can be done without leaving your current operating mode.

## Reference

**Quick test** — if you cannot complete this clearly, the objective is wrong:

> "The objective is ___, and if I achieve it, the business changes because ___."
